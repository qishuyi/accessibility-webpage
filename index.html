Accessible Components

Shuyi Qi 2025/02

This study aims at analyzing existing search box components in three commonly used applications: Chrome, PhotoShop and PowerPoint. It includes screenshots of different states of the search box components, analyzes its inputs and outputs (including for mouse/trackpad users, keyboard users, touch screen users and possibly for screen reader users), presents state diagrams of the search box component in PowerPoint and their revised versions and shares a reflection on the study.

1. Inputs

Analyzing the inputs for the search box component in Chrome, PhotoShop and PowerPoint

Chrome PhotoShop PowerPoint
Mouse/Touchpad
  • Click elsewhere outside of the search box to return to the initial state.
  • Click on the search box to go to focus state.
  • Hover over the star button or click on it.
  • Drag and drop text to copy and paste into the search box.
  • Same as for Chrome except it does not have a star button.
  • Drag and drop does not do the copying and pasting.
  • Have to right click -> “Copy” then right click -> “Paste”
  • Same as for Chrome and PhotoShop except it does not have a star button but has a cross button instead.
  • Copying and pasting works the same as PhotoShop.
  • Clicking on the cross button clears the search text.
Keyboard
  • Use tab key to exit the search box and use Command+L to return to the search box.
  • Use Command+D to click on the star button.
  • Use shift+left/right button to select or Command+A to select all and Command+C to copy. Use Command+V to paste.
  • A dropdown menu occurs while typing, Esc button removes the dropdown menu and up button restores it.
  • Use tab key to exit the search box and use the up key to return to the search box.
  • Copy and pasting works the same as for Chrome.
  • Esc button clears the text in the search box and up button leaves the search box.
  • Use Command+Ctrl+U to enter the search box and use tab key to exit.
  • Use Enter key to execute search after entering text.
  • Copy and pasting works the same as PhotoShop and Chrome.
  • Esc button works the same as PhotoShop and up button moves cursor to the leftmost position of the search text.
Touch (if available)
  • Touching the search box makes the search box focused and ready to accept text and touching elsewhere removes the focus.
  • After entering text into the search box, touching the cross icon clears the search text.
N/A
  • Touching the reading glass icon makes the search box appear and touching elsewhere outside the search box makes it disappear.
  • The cross icon works the same as Chrome.

Chrome: Default state is focused and ready to enter text when web browser is opened. Use keyboard to enter text into the search box.

PhotoShop: Default state is focused and ready to enter text when the search box page is opened. Use keyboard to enter text into the search box.

PowerPoint: Default state is not focused. Use keyboard to enter text into the search box.

The dragging and dropping for copy-and-paste functionality in Chrome is memorable, learnable and efficient because it eliminates the need for right clicks and it is more direct to memorize and easier to learn.

The Command+Ctrl+U for entering the search box in PowerPoint is learnable, memorable and efficient because this operation is encoded in the search box’s default text and it eliminates the need for moving the cursor. Using up key for returning to the search box is memorable, learnable and efficient because it is visual and intuitive, giving an impression that the search box is of the same hierarchy as all of the buttons and areas below it.

For search boxes, Chrome works the best in terms of accessibility and PhotoShop and PowerPoint are similar to each other but less accessible than Chrome.

2. Outputs

Analyzing the outputs for the search box component in Chrome, PhotoShop and PowerPoint

Chrome PhotoShop PowerPoint
Mouse/Touchpad
  • When clicked, the entire search box turns black and the “Search Google or type a URL” placeholder text disappears.
  • A dropdown of possible websites appears when user starts typing but the style of the search box itself does not change.
  • When the user types in the search box and deletes it, the background color of the search box remains black and does not restore its original color.
  • When the user types text into the search box, the star icon disappears.
  • When clicked, the borders of the search box are highlighted blue and the placeholder text does not disappear.
  • A cross icon appears when user starts typing and a dropdown appears.
  • When the cross icon is clicked on, the search box returns to the focus state.
  • Same as PhotoShop when clicked except that the borders of the search box are highlighted orange, not blue.
  • Both a cross icon and a dropdown appear when user starts typing.
  • When the cross icon is clicked on, the dropdown disappears and the search box returns to the focus state.
Keyboard
  • Before the user starts typing, when the user presses the tab key to exit the search box, the color of the search box goes back to its original color, except when the tab key selects one of the options from the dropdown and the search box is filled with text, the color of the search box remains black.
  • When the user uses Command+L to return to the search box, if there is text in the search box, the text is highlighted; otherwise there is a cursor in the search box.
  • When the user uses Command+D to click on the star button and presses Enter to select, the start button is filled with white.
  • When the user selects the text, the background of the selected text is highlighted.
  • A dropdown menu occurs as the user starts typing, pressing the Esc button removes the dropdown menu and up button restores it.
  • After user starts typing, pressing the Tab key moves down the options in the dropdown.
  • When the user exits the search box, the borders are no longer highlighted blue and when the user returns to the search box, the borders return to be highlighted.
  • Pressing the Esc button clears the text in the search box.
  • After user starts typing into the search box, pressing the Tab key moves the focus to the cross icon and the cross icon turns blue.
  • Before user starts typing, pressing the Tab key moves the cursor to other buttons.
  • The borders are highlighted orange and unhighlighted respectively when the user enters or leaves the search box using keyboard key combinations.
  • Whether the user starts typing into the search box or not, pressing tab clears the search box, makes the cross icon disappear and unhighlights the search box.
Touch (if available)
  • Touching the search box makes the background of the search box turn gray.
  • A cross icon appears when the user starts typing, touching the cross icon clears the search text.
 N/A
  • Touching the reading glass icon makes the search box appear with a cursor and the background color unchanged.
  • The cross icon works the same as Chrome.
Screen reader
  • When on a button, the screen reader reads out the functionality of that button even though it is just an icon (the bookmark icon).
  • The screen reader tells the user to use “w” key to perform an action listed as an item in the dropdown, in addition to reading out the option.
  • The screen reader informs the user of the next steps. For example, it says using up and down keys allows users to navigate items in the dropdown.
  • And when on an item in the dropdown, it says pressing Ctrl-Option-Space selects this item.

I think the logical focus order for a search box is as follows:

  • Before the user starts typing, pressing the tab moves the focus to other buttons below it.
  • After the user starts typing, pressing the tab button moves the focus to a cross button.
  • When the focus is already on the cross button, pressing the tab further moves cursor down the options in the dropdown.

  • In the revised state diagrams, the mouse/trackpad users and the keyboard users will need to perform the same action twice, once to go from the active state to the selected state and once to move from the selected state to the original state. The mouse/trackpad users need to click somewhere outside of the search box, and the keyboard users need to press the tab key.
  • When the user wants to go directly from the active state to the original state, the mouse/trackpad users need to click on the cross button while teh keyboard users need to press the Esc key.
  • By making the users perform the same action to move from active to selected and selected to original state, the revised state diagrams improve the learnability and memorability of the search boxes. And by adding the possibility for switching from the active state to the selected state, which was originally not available for keyboard users, the revised state diagram sacrificed efficiency for memorability and learnability.

5. Reflection

Reflecting on the existing search box component and my redesigned component

  • The search box for Chrome does well in usability by providing a list of search history when in active state and I incorporated it in my redesign of the PowerPoint search box.
  • The PowerPoint search box does well in accessibility by allowing user to go from active state to original state with a Tab and I kept it in the redesign.
  • When the user switches to the “Selected” state of the search box, the users will want to see the top N most recently searched keywords to refresh their memory in addition to suggestions when they start typing.
  • The current design of PowerPoint search box does not help refresh the user’s memory while they will likely need it and my redesign provides this functionality.
  • In PowerPoint, the screen reader tells the user they can press Ctrl-Option-Space to select an item in the dropdown, which benefits the users with visual impairment because it is difficult for them to click on an item using a mouse.
  • However, for the keyboard shortcut Cmd-Ctrl-U to focus on the search box in PowerPoint, it might not be possible for people without two hands because the keys are far from each other.
  • In this way it makes the design less accessible, and it will negatively impact people with impairments in their hands.
  • I think keyboard and screen reader users are most prioritized and this will lead to a more diversified and user-friendly experience of using keyboard shortcuts.
  • This choice will also make the component more accessible for people who cannot use the mouse.
